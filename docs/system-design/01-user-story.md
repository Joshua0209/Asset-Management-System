# 資產管理系統

## User Empathy Map

### Persona 1 資產持有者：入職三個月、負責客戶對接的業務助理佳慧

#### Say

- 「我的筆電螢幕突然出現條紋，請問要怎麼申請送修？」
- 「請問這次維修大概要多久？我下周有客戶簡報需要用到電腦。」
- 「上次填的維修申請，現在到哪個階段了？有沒有辦法查？」

#### Does

- 填寫紙本報修單，親自送到廠區資產管理人員的窗口提交，並將設備一同交給管理人員
- 送修後兩三天若沒有進度通知，主動透過通訊軟體詢問管理人員目前狀況
- 等到管理人員透過通訊軟體告知「設備回來了」才知道維修完成

#### Thinks

- 「萬一被認為是我弄壞的，會不會影響主管對我的評價？」
- 「設備送修期間我要怎麼工作？有沒有備用機可以借？」
- 「送修的時候，硬碟裡的資料會怎樣？萬一資料不見了，客戶的簡報檔案都沒備份。」

#### Feels

- 對只能用文字描述故障感到力不從心，擔心描述不清楚反而讓管理人員難以判斷、維修流程拖得更久
- 因無法追蹤維修進度而焦慮，不知道設備何時回來，也就無法安排自己接下來的工作
- 擔心設備送修期間硬碟資料的安全，卻不確定管理人員會不會備份，也不知道自己該不該先另外處理

---

### Persona 2 資產管理人員：與多名同事共同管理全公司七萬件設備、每人每天處理約五十筆申請的資深總務專員大偉

#### Say

- 「設備送去廠商了，但他們什麼時候修好我也不知道，使用者一直來問我怎麼辦？」
- 「每年盤點就是發問卷給全公司，叫大家自己確認設備在不在，然後我們再人工核查有沒有對不上的。」
- 「這張單我剛準備要處理，才發現同事也在跑同一張，我們每次都要靠口頭溝通才能避免重工。」

#### Does

- 在現有資產管理系統上處理報修申請，但系統狀態欄位只有簡單幾個字，詳細交接進度需另以 Teams 補充說明
- 每年盤點時發信或發問卷給全公司員工，請他們自行確認手上的設備，信任員工的回報結果
- 遇到送外部廠商維修的設備，只能要求使用者繼續等，無法給出確切的完修時間
- 與同事口頭協調報修單的分工，避免多人同時處理同一張申請造成重複作業

#### Thinks

- 「這張單現在是誰在負責？我更新了狀態，同事那邊不一定看得到，萬一兩個人同時動了同一張單怎麼辦？」
- 「一天五十張單，老闆的有優先通道，其他人就是排隊；但使用者不知道等多久，一直來追問，我也給不出準確的答案。」
- 「資產轉移的時候有沒有確實開單？如果沒走正常流程，事後要人工追查會非常麻煩。」

#### Feels

- 在需要靠口頭溝通來分工的環境裡，每一天都在用個人協調代替系統機制，既浪費時間又容易出錯，卻沒有更好的辦法
- 夾在追進度的使用者和給不出確切工期的廠商之間，什麼都解釋不清楚，有一種什麼都做了但什麼都沒解決的無力感
- 在資訊不完整、記錄靠人工維護的環境下，每一筆對不上的資產都可能開啟一場費時的人工追查，而這些工作完全不會被外部看見


## User Story

### US-01（佳慧）提交設備維修申請

身為入職三個月、從未獨立處理設備故障的業務助理佳慧，我想要透過表單填寫故障描述、上傳故障照片並提交維修申請，這樣才能在設備出問題時立刻採取正確行動，不因文字難以描述問題而被退件或在主管面前留下不負責任的印象。

### US-02（佳慧）查詢維修申請進度

身為有客戶簡報時程壓力的業務助理佳慧，我想要隨時查詢已送修設備的目前處理狀態，這樣才能提前判斷是否需要借用備用設備，不讓設備送修成為無法履行客戶承諾的意外。

### US-03（佳慧）查看個人持有設備清單

身為剛接手職務的業務助理佳慧，我想要查看登記在自己名下的所有公司設備，這樣才能在公司盤點或職務交接時提供完整準確的持有紀錄，而不是靠個人記憶憑空回想。

### US-04（大偉）審查並推進維修申請流程

身為與多名同事共同處理大量維修申請的資深總務專員大偉，我想要在系統上審查員工提交的維修申請、填寫維修內容與費用、並標記維修完成，這樣才能讓每一筆維修的處理狀態在團隊間即時同步，避免多個管理者重複處理同一張申請的溝通成本。

### US-05（大偉）查詢所有設備狀態與保固資訊

身為每年需要主導全公司資產盤點的資深總務專員大偉，我想要在系統上查詢所有設備的當前狀態、存放地點與保固到期日，這樣才能在員工回報盤點結果時有系統記錄可供人工核查比對，不再完全仰賴人工追問歷史轉移紀錄才能找出對不上的設備。

### US-06（大偉）登記新採購設備入庫

身為負責採購入庫的資深總務專員大偉，我想要在採購完成後將設備詳細資訊（型號、規格、供應商、採購金額、保固期限）登記至系統並指派給對應負責人，這樣才能讓每一件設備從入庫起就有完整可追溯的電子記錄，事後核對資產轉移或配合其他部門調資料時不需要再人工逐筆篩選。

## Acceptance Criteria (AC)

### US-01 提交設備維修申請

1. 設備持有者填寫資產編號與故障描述後，可成功提交申請，系統顯示申請成功並產生唯一申請單編號。
2. 申請提交後，申請單狀態顯示為「待審查」。
3. 設備持有者可選擇上傳故障照片，照片上傳成功後附於申請單中供管理人員查看。
4. 【邊界條件】若故障描述欄位為空，系統不允許提交，並標示必填欄位未填。
5. 【邊界條件】若輸入的資產編號不存在於系統中，系統拒絕提交並提示編號無效，要求重新輸入。
6. 【邊界條件】若該設備目前狀態已為「維修中」，系統禁止重複提交申請，並提示該設備已有進行中的維修申請。

### US-02 查詢維修申請進度

1. 設備持有者可查看自己所有歷史申請單的列表，顯示申請編號、設備名稱、提交日期與目前狀態。
2. 點入單一申請單後，可查看目前所處的流程狀態（待審查、維修中、維修完成、拒絕申請）。
3. 申請狀態由管理人員更新後，設備持有者下次查詢時即可看到最新狀態，無需重新提交申請。
4. 【邊界條件】設備持有者只能查看自己提交的申請單，無法存取其他人的申請記錄。
5. 【邊界條件】若申請狀態為「拒絕申請」，申請單顯示拒絕結果但不提供任何繼續流程的操作入口，流程已終止。

### US-03 查看個人持有設備清單

1. 設備持有者登入後可查看登記在自己名下的所有設備，顯示資產編號、名稱、型號、啟用日期與目前狀態。
2. 清單同時顯示狀態為「正常使用」與「維修中」的設備，不因設備正在維修而從清單消失。
3. 【邊界條件】若名下目前無任何設備，頁面顯示「目前無持有設備」的空狀態說明，而非空白或錯誤頁面。
4. 【邊界條件】若資產管理人員將某設備重新指派給其他人，該設備立即從原持有者的清單中移除。

### US-04 審查並推進維修申請流程

1. 資產管理人員可查看所有待審查的維修申請列表，並對每筆申請執行同意或拒絕。
2. 審查同意後，系統自動將設備狀態更新為「維修中」，申請單狀態同步更新為「維修中」。
3. 審查拒絕後，申請單狀態更新為「拒絕申請」，流程結束，設備狀態維持不變。
4. 維修進行中，資產管理人員可於申請單填寫維修日期、故障內容、維修方案、維修費用與維修人員／廠商。
5. 資產管理人員送出維修完成後，系統自動將設備狀態更新為「正常使用」，申請單狀態更新為「維修完成」，流程結束。
6. 【邊界條件】若資產管理人員嘗試對已結案的申請單（維修完成或拒絕申請）再次操作，系統拒絕並提示流程已結束。
7. 【邊界條件】送出維修完成時，若維修費用欄位為空，系統提示必填項目未完成，不允許送出。
8. 【邊界條件】若兩名資產管理人員同時開啟同一張申請單並各自嘗試更新狀態，系統僅接受先送出的一筆操作，後送出者收到提示「此申請單已由其他人員更新，請重新確認當前狀態」。

### US-05 查詢所有設備狀態與保固資訊

1. 資產管理人員可查看所有設備的完整列表，包含資產編號、名稱、型號、存放地點、負責人、狀態與保固到期日。
2. 點入單一設備後，可查看該設備的完整屬性，包含當前狀態、存放地點與保固到期日。
3. 【邊界條件】若設備的保固到期日未填寫，該欄位顯示為「未登記」，而非空白或顯示系統預設日期。
4. 【邊界條件】若系統中目前無任何設備，頁面顯示空狀態說明，而非空白頁面或錯誤訊息。

### US-06 登記新採購設備入庫

1. 資產管理人員填寫設備名稱、型號、規格、供應商、採購日期、採購金額、保固期限後，可成功新增設備，系統自動產生唯一資產編號。
2. 新增設備時可指定存放地點、負責人與使用部門；設備登記後立即出現在設備列表中，狀態預設為「正常使用」。
3. 【邊界條件】若名稱、型號、採購日期等必填欄位未填，系統不允許送出並標示缺少的欄位。
4. 【邊界條件】若保固期限填寫的日期早於採購日期，系統顯示錯誤提示，不允許儲存。
5. 【邊界條件】登記設備時若未指定負責人，系統仍允許儲存，設備列表中負責人欄位顯示為「未指派」。

## 推導說明

### 佳慧 — US-02 查詢維修申請進度

US-02 是從佳慧的 Does 與 Thinks 共同推導出來的。Does 顯示她在兩三天沒有消息後，會主動透過通訊軟體詢問管理人員；Says 也直接問出「現在到哪個階段了？」。這些表層行為揭示她並非被動等待，而是因不確定性而焦慮，必須靠主動詢問來消解壓力。從這裡推導到深層的 Feels：設備何時回來，直接決定她能不能安排借用備用機、能否準時完成手上的工作，焦慮的本質不是「想要資訊」，而是「沒有資訊就無法規劃，無法規劃就失去對工作的掌控感」。比起 US-01（送修僅發生一次）或 US-03（使用者持有設備少、查詢頻率極低），US-02 的痛點貫穿整個維修週期，對佳慧日常工作的干擾最持續，因此最值得優先處理。

### 大偉 — US-04 審查並推進維修申請流程

US-04 是從大偉的 Says、Does 與 Thinks 共同推導出來的。Says 直接描述了「才發現同事也在跑同一張單」的場景，Does 顯示他目前只能靠口頭協調來分工，Thinks 則揭示他清楚這種方式不可靠，「我更新了狀態，同事那邊不一定看得到」。從表層的重複作業推導到深層：問題不在於申請單本身難以處理，而是系統缺乏對每張單的即時歸屬機制，導致 Feels 中那種「什麼都做了但什麼都沒解決」的無力感每天反覆發生。在每人每天要處理五十張申請的規模下，若每張單都需要靠口頭確認分工，溝通成本隨量級線性放大。這個需求比 US-05 或 US-06 更值得優先處理，因為申請流程是整個維修運作的核心，流程狀態不能即時同步，其他功能再完整也無法降低管理者每天最大的摩擦。
