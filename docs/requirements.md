# 資產管理系統

## Background

公司資產種類繁多，數量龐大，且涉及採購、入庫、分配、使用、維護、盤點等多個環節。人工資產管理方式存在痛點：資訊不同步、盤點耗時耗力、資產狀況不明、維護不及時、閒置浪費等。建立一個資產管理
系統，實現資產生命週期精細化管理。

## Target Audience

- 資產持有者 (一般使用者)：提交資產維修申請，查詢個人負責的資產資訊，查詢申請進度
- 資產管理人員：負責資產的採購入庫登記、分配、維修等所有管理操作。

## System Requirement

### 資產基礎資訊管理模組

1. 資產基礎資訊
    - 資產登記與分類： 為每項資產生成唯一編號，並進行分類（如：手機類、電腦類、平板類）。
    - 資產採購入庫： 記錄資產的詳細資訊，包括名稱、型號、規格、供應商、採購日期、採購金額等。
    - 資產詳細屬性： 記錄資產的存放地點、負責人、使用部門、啟用日期、保固期限等。
    - 資產狀態：記錄資產狀態（如：正常使用、維修中）。
2. 資產查詢
    - 設備持有者可以查詢持有裝置。
    - 資產管理人員可以新增/修改/查詢所有裝置。

### 申請單管理模組

1. 資產維修管理
   - 申請流程
     - 申請：設備持有者可提交資產維修申請，填寫故障描述、資產編號等。
     - 審查：資產管理人員審查此申請
        - 審查同意：此裝置由系統更新狀態為維修中，單據狀態更新為維修中。
        - 審查拒絕：單據狀態更新為拒絕申請，此單據流程結束。
     - 維修中：資產管理人員於申請單上填寫維修的日期、故障內容、維修方案、維修費用、維修人員/廠商。
     - 維修完成：資產管理人員送出維修完成，裝置由系統更新狀態為正常使用，單據狀態更新為維修完成，此單據流程結束。
   - 申請單記錄：記錄以上提到的維修內容與流程
2. 歷史維修記錄查詢
   - 設備持有者可以查詢持有裝置申請單
   - 資產管理人員可以查詢所有裝置申請單

## Advanced Requirement

- 高可用性：系統核心服務具備容錯能力，即使部分節點故障，功能仍能正常運作。
- Zero Downtime 佈署策略
- 多語系介面支援
- 並發控制：在多用戶、高併發的環境中，如何解決多個操作可能同時讀取或修改同一份資料問題。
- 資產/申請單查詢與篩選：提供多維度（按編號、名稱、型號、地點、負責人、狀態等）的快速查詢和高級篩選功能。
- 維修申請可以上傳圖片

## Evaluation Criteria

- 30% 需求轉換與實作（是否將需求理解並轉換成 User Story，且分工並實做出來）
- 10% 程式碼品質（如何確保程式依照特定 Coding Style，且沒有 Code Smells、Security Issue 等等）
- 25% 架構設計與可擴展性（提供 Overall System Architecture Diagram、Sequence Diagram、ER Diagram 說明系統架構、技術選型，並說明擴充可能）
- 25% 系統測試與驗證（e.g. 測試覆蓋率（unit test, integration test、e2e test）、性能與穩定性測試（load test、stress test、performance test））
- 10% 運維與可靠性（e.g.,穩定性監控、告警、監控指標定義收集分析如何處理單點故障等等
